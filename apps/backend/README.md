# @ice/backend

Fastify + TypeScript API server. Thin by design — authenticates, validates,
and delegates every request to `@ice/core`'s `Brain`. Contains no business
logic (Document 2, Section 4, Rule 4).

## Running Locally

```bash
pnpm install
cp .env.example .env         # fill in real values
pnpm prisma:generate
pnpm dev
```

Server starts on `http://localhost:3000` (configurable via `PORT`/`HOST`).

## Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| GET | `/health` | none | Liveness + DB connectivity check |
| POST | `/v1/auth/register` \| `/login` \| `/refresh` | none | Proxies to Supabase Auth |
| POST | `/v1/auth/logout` | Bearer | Proxies to Supabase Auth |
| GET | `/v1/auth/me` | Bearer | Current authenticated user |
| GET/PATCH | `/v1/profile` | Bearer | User profile |
| POST/GET | `/v1/chats` | Bearer | Create/list chat sessions |
| PATCH/DELETE | `/v1/chats/:id` | Bearer | Rename/delete a session |
| GET/POST | `/v1/chats/:id/messages` | Bearer | History / send a message (calls Brain) |
| POST/GET | `/v1/projects` | Bearer | Create/list projects |
| GET/PATCH/DELETE | `/v1/projects/:id` | Bearer | Read/update/delete a project |
| POST | `/v1/files` | Bearer | Upload (multipart/form-data) |
| GET | `/v1/files/:id` \| `/:id/download` | Bearer | Metadata / raw bytes |
| DELETE | `/v1/files/:id` | Bearer | Delete (storage + metadata) |
| GET/PATCH | `/v1/settings` | Bearer | User settings (providers, routing mode) |
| POST | `/v1/memory` | Bearer | Store a memory directly |
| GET | `/v1/memory/search` | Bearer | Search memory (`?query=&types=&limit=`) |
| DELETE | `/v1/memory/:id` | Bearer | Delete a memory |
| GET | `/v1/providers` | Bearer | Provider registry + config-presence health |

### `POST /v1/chats/:id/messages`

```json
{
  "input": "your message",
  "preferredProvider": "anthropic"  // optional — omit for auto-routing
}
```

### `POST /v1/files` (multipart/form-data)

Fields: `file` (the upload), `projectId` (optional). 25MB limit enforced
server-side. Bytes go to Supabase Storage (bucket `ice-files`, must
exist — this backend doesn't create the bucket itself); metadata goes
to Postgres via the new `file_assets` table.

## Environment Variables

See `.env.example`. All variables are validated on startup by
`@ice/config` — a missing or malformed required variable fails loudly at
boot, not confusingly mid-request.

## Database Migrations

`prisma/migrations/00000000000000_init/migration.sql` is hand-authored
to match `schema.prisma` — it was not generated by `prisma migrate dev`
because this development environment has no network access to a live
database. Before relying on it:

```bash
pnpm --filter @ice/backend prisma migrate dev --name init
```

against a real Supabase/Postgres instance, and let Prisma verify (or
regenerate) it from the schema directly. You'll also need to manually
create a Supabase Storage bucket named `ice-files` — this backend
uploads to it but doesn't provision it.

## Adding a Route

1. Create `src/routes/your-thing.route.ts` — auth (handled by the parent
   router group) → zod validation → one call into `container.brain` (or a
   new Core entry point you've added) → serialize response.
2. Register it in `src/routes/index.ts`.
3. Never put business logic in the route handler itself — see
   `docs/architecture/02-system-architecture.md`, Section 7.2.
